# TECH_DESIGN

## 1. 技术栈选择

本项目是一个纯前端个人待办事项页面，主要实现任务的添加、展示、完成、删除、编辑、筛选和优先级区分。

项目整体保持简单，不使用复杂框架，也不依赖后端服务。

### 1.1 前端用什么

| 技术 | 作用 |
|---|---|
| HTML | 搭建页面结构 |
| CSS | 实现页面样式和布局 |
| JavaScript | 实现页面交互和任务逻辑 |
| localStorage | 保存本地任务数据 |

### 1.2 后端用什么

MVP 阶段不使用后端。

原因：

- 项目是个人待办事项页面
- 不需要用户登录
- 不需要多人协作
- 数据量较小
- 使用浏览器本地存储即可满足需求

后续如果需要登录、云端同步、多设备使用，可以再增加后端接口。

### 1.3 数据库用什么

MVP 阶段不使用数据库。

任务数据直接保存在浏览器的 `localStorage` 中。

后续如果需要服务端保存，可以考虑：

| 数据库 | 说明 |
|---|---|
| MySQL | 适合结构化任务数据 |
| SQLite | 适合轻量级项目 |
| MongoDB | 适合 JSON 形式数据 |

## 3. 数据模型

本项目需要存储待办事项数据。每条任务使用一个对象表示，所有任务使用数组保存。

### 3.1 任务数据结构

```js
{
  id: 1,
  title: "完成待办事项页面",
  completed: false,
  priority: "medium",
  createdAt: "2026-05-18 10:00:00"
}
```

### 3.2 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Number | 任务唯一编号 |
| title | String | 任务标题 |
| completed | Boolean | 是否完成 |
| priority | String | 任务优先级 |
| createdAt | String | 创建时间 |

### 3.3 优先级字段说明

| priority 值 | 含义 |
|---|---|
| high | 高优先级 |
| medium | 中优先级 |
| low | 低优先级 |

### 3.4 本地存储方式

任务列表通过 `localStorage` 保存。

保存数据：

```js
localStorage.setItem("todoList", JSON.stringify(todoList));
```

读取数据：

```js
const todoList = JSON.parse(localStorage.getItem("todoList")) || [];
```

## 4. 关键技术点

### 4.1 快速添加任务

添加任务是页面最核心的功能。

实现要求：

- 页面顶部显示明显的输入框
- 输入内容不能为空
- 点击添加按钮可以添加任务
- 按 Enter 键也可以添加任务
- 添加后清空输入框
- 添加后保存到 localStorage
- 添加后重新渲染任务列表

基本流程：

```text
输入任务内容
↓
校验内容是否为空
↓
创建任务对象
↓
加入任务数组
↓
保存到 localStorage
↓
重新渲染页面
```

---

### 4.2 任务列表渲染

任务列表由 JavaScript 根据任务数组动态生成。

每条任务显示：

- 完成复选框
- 任务标题
- 优先级标签
- 编辑按钮
- 删除按钮

需要注意：

- 已完成任务需要显示删除线
- 高、中、低优先级要有不同样式
- 没有任务时显示空状态提示
- 不要让按钮太抢眼，避免影响页面简洁性

---

### 4.3 任务完成状态切换

用户点击复选框后，修改任务的完成状态。

实现要求：

- 点击后切换 `completed` 值
- `completed` 为 `true` 时，任务显示为已完成
- 已完成任务文字变灰并添加删除线
- 状态变化后保存到 localStorage

---

### 4.4 删除任务

用户点击删除按钮后，删除对应任务。

实现要求：

- 根据任务 `id` 找到对应任务
- 从任务数组中移除
- 删除后重新渲染页面
- 删除后更新 localStorage

---

### 4.5 编辑任务

用户可以修改任务标题。

实现要求：

- 点击编辑按钮后进入编辑状态
- 用户可以修改任务内容
- 修改后的内容不能为空
- 保存后更新任务标题
- 保存后更新 localStorage

---

### 4.6 任务筛选

页面支持简单筛选。

筛选项包括：

| 筛选项 | 说明 |
|---|---|
| 全部 | 显示所有任务 |
| 未完成 | 只显示未完成任务 |
| 已完成 | 只显示已完成任务 |

实现要求：

- 点击筛选按钮后切换当前筛选状态
- 当前筛选项需要高亮显示
- 筛选只影响展示，不删除真实数据

---

### 4.7 优先级视觉区分

任务需要体现轻重缓急。

| 优先级 | 样式建议 |
|---|---|
| 高 | 红色标签或红色边框 |
| 中 | 蓝色或橙色标签 |
| 低 | 灰色标签 |

实现方式：

- 给不同优先级任务添加不同 class
- 使用 CSS 设置不同颜色
- 高优先级任务可以排在前面

示例：

```html
<li class="todo-item priority-high">
  <span class="priority-tag">高</span>
  <span>完成页面开发</span>
</li>
```

---

### 4.8 本地存储

因为项目没有后端，所以需要使用 `localStorage` 保存任务数据。

需要在这些操作后保存数据：

- 添加任务
- 删除任务
- 编辑任务
- 完成任务
- 取消完成任务

需要注意：

- localStorage 只能保存字符串
- 数组需要通过 `JSON.stringify` 转成字符串
- 读取时需要通过 `JSON.parse` 转回数组
- 如果本地没有数据，默认返回空数组

## 5. 技术难点和注意事项

### 5.1 页面要清晰干净

本项目重点是待办事项本身，不能让功能压过页面核心业务。

需要避免：

- 功能太多
- 按钮太多
- 颜色太杂
- 弹窗太复杂
- 页面信息堆叠太多

建议：

- 添加区域明显
- 任务列表简洁
- 操作按钮适当弱化
- 优先级颜色清楚但不过度夸张

---

### 5.2 用户输入安全

任务标题是用户输入内容，需要避免安全问题。

注意事项：

- 不要直接使用 `innerHTML` 渲染用户输入
- 推荐使用 `textContent` 显示任务标题
- 输入内容需要使用 `trim()` 去除前后空格
- 空内容不能提交

---

### 5.3 数据保存问题

使用 localStorage 时需要注意：

- 页面加载时要先读取本地数据
- 每次修改任务后都要保存
- 数据读取失败时要有默认空数组
- localStorage 不适合保存大量数据

---

### 5.4 任务 id 生成

每个任务都需要唯一 id，方便删除、编辑和修改状态。

简单实现方式：

```js
const id = Date.now();
```

---

### 5.5 排序规则

为了体现轻重缓急，可以对任务进行简单排序。

推荐规则：

```text
未完成任务在前
高优先级任务在前
中优先级任务其次
低优先级任务最后
已完成任务靠后
```

---

### 5.6 响应式适配

页面需要在电脑和手机上都能正常使用。

注意事项：

- 主体内容居中
- 移动端输入框和按钮可以上下排列
- 任务标题过长时允许换行
- 按钮点击区域不能太小

---

## 6. 总结

本项目采用纯前端方式实现，推荐技术栈为：

```text
HTML + CSS + JavaScript + localStorage
```

MVP 阶段重点实现：

- 快速添加任务
- 展示任务列表
- 完成任务
- 删除任务
- 编辑任务
- 任务筛选
- 优先级视觉区分
- 本地数据保存

开发重点：

- 添加入口明显
- 页面清晰干净
- 功能简单够用
- 不让功能压过任务本身
- 高、中、低优先级有明显区分
