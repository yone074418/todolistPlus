# 个人待办事项页面 PRD

## 1. 产品概述和目标用户

### 1.1 产品概述

本产品是一个纯前端的个人待办事项页面，用于帮助用户快速记录、查看和管理日常任务。

页面设计以“清晰、干净、简单”为核心，不追求复杂功能，重点突出待办事项本身，让用户可以快速添加任务、查看任务状态，并区分任务的轻重缓急。

### 1.2 产品目标

- 支持用户快速添加待办事项
- 添加任务入口明显，操作简单
- 页面结构清晰，内容干净
- 不让过多功能影响用户关注任务本身
- 支持任务完成、删除和简单编辑
- 支持任务优先级区分
- 使用本地存储保存数据，刷新页面后数据不丢失

### 1.3 目标用户

| 用户类型 | 使用场景 |
|---|---|
| 学生 | 记录作业、考试、学习计划 |
| 上班族 | 记录每日工作任务 |
| 普通个人用户 | 管理生活、学习、工作中的待办事项 |
| 前端学习者 | 作为纯前端项目练习和展示 |

## 2. 功能列表

### 2.1 快速添加任务

#### 功能说明

页面需要提供一个明显的任务添加区域，用户可以快速输入任务内容并添加到任务列表中。

#### 功能要求

- 页面顶部显示任务输入框
- 输入框提示文字为：`添加一个新的待办事项`
- 用户输入内容后，点击“添加”按钮即可创建任务
- 支持按 `Enter` 键快速添加任务
- 不允许添加空任务
- 添加成功后，输入框自动清空

---

### 2.2 任务列表展示

#### 功能说明

用户添加的任务需要以列表形式展示，方便用户查看当前所有待办事项。

#### 每条任务包含内容

- 完成状态
- 任务标题
- 任务优先级
- 删除按钮
- 编辑按钮

#### 展示规则

- 未完成任务正常显示
- 已完成任务文字变灰，并添加删除线
- 高优先级任务需要明显突出
- 中优先级任务正常显示
- 低优先级任务弱化显示

---

### 2.3 设置任务优先级

#### 功能说明

用户可以给任务设置优先级，用于区分任务的重要程度。

#### 优先级类型

| 优先级 | 含义 | 视觉表现 |
|---|---|---|
| 高 | 重要，需要优先完成 | 红色标签或红色边框 |
| 中 | 普通任务 | 蓝色或橙色标签 |
| 低 | 不紧急任务 | 灰色标签 |

#### 功能要求

- 添加任务时可以选择优先级
- 默认优先级为“中”
- 不同优先级需要有明显的视觉区分
- 高优先级任务可以排在任务列表前面

---

### 2.4 完成任务

#### 功能说明

用户可以将任务标记为已完成，也可以取消完成状态。

#### 功能要求

- 每条任务前面有复选框
- 点击复选框后，任务变为已完成
- 已完成任务文字变灰
- 已完成任务标题显示删除线
- 再次点击复选框，可以取消完成状态

---

### 2.5 删除任务

#### 功能说明

用户可以删除不需要的任务。

#### 功能要求

- 每条任务提供删除按钮
- 点击删除后，该任务从列表中移除
- 删除后页面立即更新
- 删除按钮不要过于突出，避免影响页面简洁性

---

### 2.6 编辑任务

#### 功能说明

用户可以修改已经添加的任务内容。

#### 功能要求

- 每条任务提供编辑按钮
- 点击编辑按钮后，可以修改任务标题
- 修改完成后可以保存
- 可以取消编辑
- 编辑功能保持简单，不需要复杂弹窗

---

### 2.7 任务筛选

#### 功能说明

用户可以根据任务状态筛选任务。

#### 筛选项

| 筛选项 | 说明 |
|---|---|
| 全部 | 显示所有任务 |
| 未完成 | 只显示未完成任务 |
| 已完成 | 只显示已完成任务 |

#### 功能要求

- 筛选按钮放在任务列表上方
- 当前选中的筛选项需要高亮显示
- 切换筛选项时，任务列表实时更新

---

### 2.8 本地数据保存

#### 功能说明

由于本项目是纯前端页面，因此需要使用浏览器本地存储保存任务数据。

#### 功能要求

- 使用 `localStorage` 保存任务数据
- 页面刷新后任务数据不丢失
- 添加任务后自动保存
- 删除任务后自动保存
- 修改任务后自动保存
- 完成任务后自动保存

---

### 2.9 空状态提示

#### 功能说明

当任务列表为空时，需要显示友好的提示内容。

#### 示例文案

暂无待办事项，添加一个任务开始规划今天吧。

#### 功能要求

- 页面不能完全空白
- 提示用户可以添加新任务
- 空状态设计保持简洁

---

## 4. 界面设计要求

### 4.1 整体风格

页面整体风格需要保持：

- 简洁
- 干净
- 清晰
- 操作直观
- 重点突出任务内容

不要使用过多颜色、复杂图标和大量动画，避免影响用户关注任务本身。

---

### 4.2 页面布局

推荐页面结构如下：

```text
页面标题

添加任务区域
[ 输入框 ] [ 优先级选择 ] [ 添加按钮 ]

筛选区域
全部 / 未完成 / 已完成

任务列表区域
□ 高优先级任务
□ 普通任务
✓ 已完成任务

空状态提示
暂无待办事项
```

---

### 4.3 添加任务区域

添加任务区域是页面的核心入口，需要足够明显。

#### 设计要求

- 放在页面顶部或主内容区域显眼位置
- 输入框宽度较大
- 添加按钮清楚可见
- 输入框和按钮之间保持合理间距
- 支持键盘快速添加任务

---

### 4.4 任务列表设计

任务可以使用列表项或卡片形式展示。

#### 每条任务建议结构

- 左侧：完成复选框
- 中间：任务标题
- 右侧：优先级标签、编辑按钮、删除按钮

#### 设计要求

- 每条任务之间留有适当间距
- 任务内容清楚易读
- 操作按钮不要过于抢眼
- 已完成任务需要和未完成任务明显区分

---

### 4.5 优先级视觉区分

不同优先级需要有明显但不过度夸张的视觉表现。

| 优先级 | 样式建议 |
|---|---|
| 高 | 红色标签、红色左边框 |
| 中 | 蓝色或橙色标签 |
| 低 | 灰色标签 |

高优先级任务可以更加突出，但不能让页面显得杂乱。

---

### 4.6 响应式设计

页面需要适配不同屏幕尺寸。

#### 要求

- 桌面端内容居中显示，宽度适中
- 移动端输入框和按钮可以上下排列
- 小屏幕下任务内容仍然清楚可读
- 按钮点击区域不能太小

---

## 5. 技术栈建议

### 5.1 基础技术栈

适合简单纯前端项目：

| 技术 | 用途 |
|---|---|
| HTML | 页面结构 |
| CSS | 页面样式 |
| JavaScript | 页面交互和数据处理 |
| localStorage | 本地数据保存 |

---

### 5.2 推荐实现方式

如果只是做简单页面，推荐使用：

```text
HTML + CSS + JavaScript + localStorage
```

如果想让项目结构更规范，可以使用：

```text
Vue 3 + Vite + localStorage
```

或：

```text
React + Vite + localStorage
```

---

### 5.3 数据结构建议

任务对象可以设计为：

```js
{
  id: 1,
  title: "完成前端页面设计",
  completed: false,
  priority: "high",
  createdAt: "2026-05-18 10:00:00"
}
```

#### 字段说明

| 字段 | 说明 |
|---|---|
| id | 任务唯一编号 |
| title | 任务标题 |
| completed | 是否完成 |
| priority | 优先级 |
| createdAt | 创建时间 |

## 6. 非功能性需求

### 6.1 性能需求

- 页面加载速度要快
- 添加、删除、完成任务时不能明显卡顿
- 任务数量在 100 条以内时应保持流畅
- 避免使用过大的图片和复杂动画
- 尽量减少不必要的 DOM 操作

---

### 6.2 安全需求

由于任务内容由用户输入，需要注意基础安全问题。

#### 要求

- 不直接使用 `innerHTML` 渲染用户输入内容
- 防止用户输入恶意脚本
- 用户输入内容需要进行简单过滤或转义
- 数据只保存在本地，不上传服务器

---

### 6.3 可用性需求

- 页面操作简单，新用户无需学习即可使用
- 添加任务入口明显
- 按钮文字清楚，例如“添加”“编辑”“删除”
- 空状态需要有提示
- 空任务不能添加，并给出提示

---

### 6.4 兼容性需求

页面应支持主流现代浏览器：

- Chrome
- Edge
- Firefox
- Safari

移动端浏览器也应基本可用。

---

### 6.5 可维护性需求

代码结构要清晰，方便后续扩展。

#### 建议

- HTML 结构清楚
- CSS 样式命名规范
- JavaScript 功能拆分明确
- 任务数据统一管理
- 关键逻辑添加必要注释
